# ShelfFrame

## 1. 项目概述

这是一个本地优先、数据库驱动、带网页前端的个人媒体库系统，用于统一管理本地漫画与视频资源。支持扫描本地文件夹、维护元数据、标签分类、多层搜索、网页阅读/播放，以及进度记录等能力。项目目标不是做一个单纯的文件浏览器，而是做一个可长期使用的本地媒体资料管理系统。

---

## 2. 产品定位

### 2.1 产品目标

为个人收藏场景提供一个统一的本地媒体库，解决这些问题：

* 本地漫画和视频散落在不同目录，难以整理
* 文件名混乱，难以查找
* 缺少标签、分类、搜索能力
* 看过哪里、读到哪里没有统一记录
* 浏览体验差，缺少封面、简介、章节信息、预览图等

### 2.2 目标用户

当前以**单用户个人使用**为主，默认运行在本机，也可扩展为局域网访问。

### 2.3 非目标

第一阶段不以这些为目标：

* 在线资源聚合
* 云同步
* 多用户权限系统
* 视频转码服务
* 商业化平台能力
* 自动抓取外站完整资料库

## 3. 已确认的核心产品决策

### 3.1 数据主导方式

采用 **方案 B：数据库为准**。

含义是：

* “作品”“标签”“简介”“章节标记”“播放进度”等逻辑实体，以数据库为主
* 文件系统是资源来源，不是唯一真相来源
* 文件变化会触发数据库更新，但不会完全覆盖数据库中的人工整理结果

### 3.2 元数据存储方式

采用 **方案 B：数据库 + sidecar 文件**。

约定：

* 运行时以数据库为主
* 每个作品目录旁边维护一个 `metadata.json`
* `metadata.json` 用于备份、迁移、人工编辑、跨设备保留整理成果
* 数据库与 sidecar 冲突时，默认**数据库优先**
* 可在设置中提供“导入 sidecar 覆盖数据库”或“导出数据库到 sidecar”的显式操作

### 3.3 漫画组织规则

采用：**文件夹视为作品**。

约定：

* 一个作品目录对应一个漫画作品
* 作品目录下可以包含图片、子目录、压缩包等资源
* 子目录可识别为章节/卷/话
* 目录是漫画归属与 sidecar 的基本边界

### 3.4 视频播放约束

目标要求：**尽可能支持所有编码，且不做转码**。
这里需要明确一个技术事实：

> 纯网页播放器无法在“不转码”的前提下真正支持所有编码，因为最终是否能播放取决于浏览器和系统解码能力。

所以工程上建议写成：

* 产品目标：尽可能兼容更多本地视频编码
* 默认策略：浏览器原生播放
* 补充策略：对浏览器不支持的编码，提供“外部播放器打开”或“本地原生辅助层”方案
* 明确第一阶段**不做服务端转码**

### 3.5 搜索能力目标

搜索分三层：

* **基础搜索**：标题、标签、文件名字符串匹配
* **增强搜索**：拼音、别名、模糊匹配、分词
* **高级搜索**：查询语法

### 3.6 视频播放器目标

播放器除了常规能力外，还应支持：

* 播放 / 暂停 / 全屏 / 拖拽 / 音量 / 快捷键
* 齿轮菜单中的播放速度
* 进度条上方可编辑的章节标签
* 章节切分与时间轴信息
* 短标签显示
* description 等元数据展示
* 热度图叠加
* 预览图（hover thumbnail）

## 4. 产品形态与核心体验

系统由三部分组成：

* **本地后端服务**：扫描文件、维护数据库、管理索引、生成缩略图和预览数据
* **网页前端**：作品浏览、搜索、阅读、播放、编辑元数据
* **本地文件系统**：漫画目录、视频目录、sidecar 元数据

用户的核心操作链路是：

1. 配置本地目录
2. 系统扫描入库
3. 自动生成作品条目和缩略图
4. 用户在网页中浏览、搜索、打标签
5. 用户阅读漫画、播放视频
6. 系统记录进度、最近访问、历史行为
7. 用户可编辑作品资料并同步到数据库 / sidecar

## 6. 详细需求说明

## 6.1 媒体扫描与入库

### 扫描对象

* 漫画：目录、图片集合、可扩展支持压缩包
* 视频：单视频文件

### 扫描规则

* 漫画：文件夹视为作品
* 视频：单文件视为作品；若后续支持“剧集文件夹”，可扩展为系列模型
* sidecar：若同目录存在 `metadata.json`，则读取并尝试合并

### 识别字段

* 路径
* 文件名
* 文件大小
* 扩展名
* 修改时间
* 哈希 / 指纹
* 封面候选
* 章节结构

### 同步策略

* 新文件：新建资源与作品
* 文件缺失：标记失效，不直接删除作品
* 文件移动：尝试基于哈希或目录指纹重关联
* sidecar 变化：记录“待同步状态”，避免静默覆盖数据库

---

## 6.2 数据同步原则

这是数据库为准方案下最重要的一条。

### 运行时优先级

`数据库 > sidecar > 文件系统推断`

### 同步规则建议

1. 新导入作品时
   文件扫描结果 + sidecar 一起建档

2. 用户在网页中编辑时
   先写数据库，再按策略写回 sidecar

3. 下次扫描时
   若发现 sidecar 比数据库新，不自动覆盖，标记为“有冲突”

4. 提供三种冲突处理动作

   * 以数据库覆盖 sidecar
   * 以 sidecar 覆盖数据库
   * 手动合并

## 7. 非功能需求

## 7.1 性能

* 首屏作品列表应快速加载
* 扫描应支持增量，避免每次全量重建
* 搜索响应应控制在可接受范围内

* 文件缺失不应导致作品直接消失
* sidecar 冲突不能静默覆盖
* 数据库损坏应支持备份恢复

## 7.3 可维护性

* 搜索、扫描、播放器、阅读器模块独立
* sidecar schema 可版本化
* API 与前端解耦
* 数据模型具备后续扩展空间

## 7.4 可扩展性

* 可扩展更多文件类型
* 可扩展系列 / 多季 / 多卷模型
* 可扩展外部播放器与原生辅助层

## 8.2 sidecar 示例结构

`metadata.json` 建议至少包含：

```json
{
  "schema_version": 1,
  "work_id": "uuid",
  "type": "comic",
  "title": "作品标题",
  "aliases": ["别名1", "别名2"],
  "description": "作品简介",
  "tags": ["热血", "冒险"],

    {
      "title": "第1话",
      "index_no": 1
    }

  "custom": {}
```

```json
{
  "timeline": {
    "chapters": [
      { "title": "开场", "start": 0, "end": 85, "short_label": "OP" }
    ]
  }
}
```

## 9. 技术架构草图

## 9.1 推荐技术路线

### 前端

建议：**Vue 3 + TypeScript**
原因：

* 播放器、阅读器、复杂状态管理更容易组织
* 搜索页、详情页、编辑页、时间轴交互都适合 Vue 的组合式 API 与组件化模式

可选组件：

* 路由：Vue Router
* 状态管理：Pinia
* UI：Tailwind + 组件库
* 视频播放器层：原生 video + 自定义控制层

### 后端

建议：**Python + FastAPI**

原因：

* 本地文件扫描、媒体处理、任务编排方便
* 调 ffmpeg / ffprobe 很顺手
* sidecar 读写和索引重建也容易做

### 数据库

建议：**SQLite**
原因：

* 本地项目最轻量
* 易部署、易备份
* 足够支持个人媒体库场景

### 搜索

建议分两阶段：

* V1：SQLite FTS + 规范化字段
* V2：抽象 SearchAdapter，可切换更强搜索引擎

### 媒体处理

* `ffprobe`：读取视频元数据
* `ffmpeg`：生成封面、预览图、缩略图
* 图片处理：Pillow
* 文件监听：watchdog

## 10. 关键风险与设计提醒

## 10.1 “支持所有编码但不转码”的风险

这是当前最大的技术约束。

结论是：

* **纯 Web 前端无法保证所有编码都可播放**
* 可以做到“尽可能兼容”
* 对于不支持的格式，建议提供这两种兜底方式：

  1. 调起系统外部播放器
  2. 后续增加本地原生辅助层

这条建议应该直接写进项目 README，避免目标和现实不一致。

## 10.2 数据库优先 + sidecar 并存的复杂度

这会带来冲突管理问题。
必须尽早定义同步规则，否则后面很容易出现“谁覆盖谁”混乱。

## 10.3 搜索能力越强，越要抽象

拼音、模糊、别名、分词、高级语法最好不要硬写死在数据库层，建议抽象成独立搜索模块。

## 10.4 视频时间轴能力会拉高工程复杂度

预览图、热度图、章节标签这几个功能都不是单纯前端功能，需要后端提前准备数据。

## 13. 一句话架构结论


> **Vue 网页前端 + FastAPI 本地服务 + SQLite 元数据库 + sidecar 元数据文件 + ffmpeg 媒体处理 + 可扩展搜索模块** 的本地优先媒体库系统。

## 6. 技术栈落地决策（已更新）

为匹配当前开发阶段，项目骨架采用以下技术栈：

* **后端**：Python + FastAPI
* **数据库**：SQLite + SQLAlchemy
* **前端**：Vue 3 + Vite
* **扫描能力**：后端本地文件系统扫描服务
* **阅读器 / 播放器**：前端先实现原型页面，后续逐步增强交互和媒体能力

说明：

* 该技术栈与 README 既有目标不冲突，可直接支持本地优先、数据库驱动、Web 前端的产品方向
* 首阶段优先把“可运行骨架”搭起来，再逐步补齐 sidecar、标签、搜索增强、预览图、章节时间轴等能力

## 9. 启动方式（骨架阶段）

### 9.1 后端

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cd backend
uvicorn app.main:app --reload
```

### 9.2 前端

```bash
cd frontend
npm install
npm run dev
```

默认情况下：

* 后端运行在 `http://127.0.0.1:8000`
* 前端运行在 `http://127.0.0.1:5173`
* Vite 已代理 `/api` 到本地后端
